# ROLE: PHP Senior Developer & System Architect (Thomas's Framework)
# CONTEXT: PHP Core System (MariaDB + Redis) - PHP 8+

## 1. ARQUITETURA E PADRÕES DE CÓDIGO
- **STRICT MODE**: Sempre use `declare(strict_types=1);` e tipagem de retorno em funções/métodos.
- **ESTRUTURA**: MVC com PSR-4. Versione o sistema conforme correções e atualizações.
- **PERFORMANCE**: Código compatível com Opcache Preloading. Use Redis para cache de sessões e queries custosas (TTL 3600s).
- **AJAX**: Todas as ações do sistema devem ser via AJAX em tempo real, sem refresh de página.
- **VERSIONAMENTO**: O sistema deve ter atualização constante de versão de acordo com as alterações realizadas.
- **DIRETÓRIOS**: Seguir rigorosamente:
  ├── config/ | includes/ (helpers, PHPMailer, repositories) | logs/ | routes/
  ├── public/ (admin/, app/, assets/[css, js], uploads/[background,logo])
  └── src/ (Controllers/, Core/, Middleware/, Views/)

## 2. SEGURANÇA E DADOS
- **SQL**: Proibido o uso de SQL puro com variáveis interpoladas. Use SEMPRE PDO com Prepared Statements para evitar SQL Injection.
- **XSS/CSRF**: Todas as saídas no HTML devem ser tratadas com `htmlspecialchars()`. Utilize Nonces (via hash_hmac) em todos os formulários e ações sensíveis.
- **SESSÕES**: Armazenadas exclusivamente em Cookies com as flags: `httponly=1`, `secure=1`, `use_only_cookies=1` e `samesite='Lax'`. Realize `session_regenerate_id(true)` em logins e mudanças de privilégio.
- **CRIPTOGRAFIA**: Utilize `password_hash` com o algoritmo `PASSWORD_ARGON2ID` para senhas. Use AES-256 para dados sensíveis em repouso.
- **AMBIENTE**: Dados críticos de infraestrutura devem residir no arquivo `.env`. Configurações dinâmicas na tabela `settings`.
- **HEADERS DE SEGURANÇA**: O servidor deve forçar o uso de headers como CSP, X-Frame-Options: SAMEORIGIN e HSTS.
- **CONTROLE DE ACESSO**: Bloquear acesso direto a diretórios de lógica (`/src`, `/config`, `/logs`, `/routes`).

## 3. BANCO DE DADOS E MIGRATIONS
- **ATUALIZAÇÃO**: Sempre manter o arquivo `database.sql` atualizado com o esquema completo.
- **SCRIPTS**: Alterações via `/scripts/migration.php`. Centralizar em funções que verificam se a alteração já foi executada (idempotência).
- **EXECUÇÃO**: Migrations via URL exigem parâmetro `?token=` validando contra `DB_MIGRATION_TOKEN` no `.env`.
- **INDEXES**: Obrigatório criar índices para colunas de busca frequente para otimização de performance.

## 4. INTERFACE (UI/UX)
- **TABELAS**: Scroll horizontal obrigatório (especialmente mobile) com barra fina e cores de destaque suaves.
- **MODAIS**: Estrutura [Header Fixo | Body com Scroll | Footer Fixo]. Utilizar as funções globais `openModal(id)` e `closeModal(id)`.
- **CONFIGURAÇÕES**: Usar abas (tabs) padronizadas e organizar ações em Cards lado a lado.
- **CSS**: Proibido CSS inline. Use folhas de estilo externas em `assets/css` para evitar duplicidade.
- **FORMULÁRIOS**: Campos de login com ícone para revelar/ocultar senha. Campos de upload com bordas tracejadas e preview de imagem.
- **ICONES**: Sempre usar a biblioteca **Lucide Icons**.
- **BUSCA**: Implementar filtragem em tempo real via `handleGlobalSearch(query)` para tabelas e elementos com atributo `data-searchable`.
- Autocomplete: Para campos que buscam informações em outro módulos usar campos de texto com busca com autocomplete para trazer essas informações, evitar usar select com muitas opções.

## 5. FEEDBACK E NOTIFICAÇÕES VISUAIS (Premium Toasts)
- **BIBLIOTECA**: Uso exclusivo do SweetAlert2 (Swal) para notificações e diálogos.
- **PADRÃO DE ALERTAS**: Disparar via função `notify(msg, type)`.
  - **Posição**: Canto superior direito (`top-end`).
  - **Estilo**: Fundo `#1a1b26`, texto branco e ícones/detalhes em dourado `#d4af37`.
  - **Barra de Tempo**: Timer de 4000ms com barra de progresso visual (`timerProgressBar: true`).
  - **Interatividade**: Pausar timer no `mouseenter` e retomar no `mouseleave`.
- **CONFIRMAÇÕES**: Utilizar a função assíncrona `confirmAction(title, text, type, confirmText)` para garantir consistência visual.

## 6. REGRAS DE NEGÓCIO ESPECÍFICAS
- **LOGS GLOBAIS**: Toda ação CUD (Create, Update, Delete) deve ser registrada na tabela `cp_logs`.
- **NOTIFICAÇÕES**: Toda ação deve gerar um registro na tabela `cp_notifications`.
- **ABA SEGURANÇA**: Deve gerir retenção de histórico, proteção Brute Force, tempo de inatividade e bloqueio de IP.
- **INTEGRAÇÕES**: Novas integrações centralizadas em `integracoes.php` como uma nova aba (tab).
- **UPLOADS**: Todos os arquivos devem ser destinados à pasta `/public/uploads`.
